01

Separate accent and warn hues.

§7 trust-led · §11 state semantics

Defect confirmed. v0.1 accent #C9A24B and warn #D4A24B share the same hue family (yellow/ochre, H≈44°) and near-identical chroma. On --ink-150 the two colors register a 1.04:1 contrast ratio against each other — indistinguishable at the 10px badge sizes where warn lives. That collapses "winner / highlight" into "warning / dispute" and violates §7 trust-led semantics.

Resolution recommendation: shift warn away by both hue and chroma — not luma alone — so the two remain legibly separate even at 10px badge + small-swatch scale. Three candidates evaluated below; RECOMMEND rust #D97757.

04

Product assumptions, flagged.

Pending product confirmation before Phase 2 sign-off

Four decisions that belong to product, not design — each marked ASSUMED · confirm with the rationale we used. The design will carry these into Phase 2; a product decision to shift any of them changes component configuration, not system structure.

Default leaderboard filters
ASSUMED · confirm
category = All · timeframe = 90d · tier = All · sample-size floor N ≥ 10
Rationale. 90d balances recency and statistical substance; a tighter window (30d) amplifies noise contrary to §9 freshness vs noise. N≥10 matches §4.1 "credible sample size" without eliminating new entrants. Product to confirm before launch copy and first-visit telemetry are finalized. §5A · §8
Sample-size floor (sparse-data threshold)
ASSUMED · confirm
Low N badge fires when N < 20 in selected window · score attenuated below N < 10
Rationale. Two-tier threshold so "insufficient to rank confidently" and "insufficient to score at all" read differently. Below N=10 the score renders at 60% opacity with a confidence-interval tooltip §13 low-confidence. Product/data-science to confirm against actual creator-call distributions. §13 · §14
"New / recently updated" TTL
ASSUMED · confirm
24 hours from data arrival · 4px left-border in --new · no animation
Rationale. 24h is long enough that a user visiting daily sees a cohort of "new" markers, short enough that the signal does not dilute. If creator universe posts at markedly higher frequency than expected, drop to 12h. §13
Free vs paid · state preservation ÷ saved views
ASSUMED · confirm
Free: URL + localStorage state persistence (last session) · Paid: named, shareable saved views + cross-device sync
Rationale. Persisting last session on free is table-stakes usability and does not undercut saved views as a paid feature — saved views' value is naming, sharing, and multi-view parallel analysis §15. Hard-gating basic state preservation would create §9 gating vs frustration without value. Product to confirm. §15 · §16.4
05

Deferrals · out of scope for Phase 2.

Explicitly noted so gaps aren't mistaken for omissions
DEFERRED · PHASE 3 (if scheduled)
Light theme is deferred.
Per §11: "a light theme is secondary priority." Token architecture is designed to invert — every semantic token references an ink step, not a hex; a light-theme ramp can be substituted without component rewrites. In-phase work: none. Phase 3 scope: invert ramp, re-audit state colors on paper, verify chart palette on #F4F5F7 ground.
DEFERRED · PHASE 2 MOBILE PASS
Mobile type scale is deferred · desktop-first this phase.
Per §17: mobile is scoped, not parity. Phase 2 produces desktop components at the current type scale; mobile responsive variants (§18.7) come as a subsequent pass with their own scale (no table row <28px on touch; body ≥14px; hit targets ≥44px). Do not read "no mobile specimen" as "mobile forgotten." In-phase work: components will declare mobile-substitution rules inline.
